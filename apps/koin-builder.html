The Builder's Guide · Implementing Koin · v1.0

The Builder's Guide

A concrete protocol specification for the person who wants to build a Koin pilot. Domain-agnostic. Engineer-readable. Plain-language for each technical concept.

This document is for the engineer, the small-team founder, the public-sector technologist, or the curious developer who has read the manifesto and now wants to build something. It does not assume that you have read the manifesto in detail; it does assume that you understand the concept of an influence graph and the idea of routing payments to upstream contributors.

The protocol described here is the first reference specification of Koin. It is intentionally minimal — enough to ship a small pilot in a single domain, with the option to federate later. Every component is described twice: once in plain English so the non-engineer can verify it, once in implementation-ready detail so the engineer can build it.

If you build a Koin pilot using this guide, your implementation will be compatible with the open federation specification (when the federation specification is finalized). If you build a Koin-like system without following this guide, you may end up with a closed implementation that cannot federate — which is the outcome the manifesto warns against.

I — Architecture overview

A Koin implementation consists of seven components. They can be deployed on a single machine for a small pilot or distributed across many servers for production.

Identity registry. The system that maintains the identifiers of contributors. Each contributor has a did:koin DID that resolves to public-key material and chosen payment substrates. The identity registry is decentralized — no central authority issues DIDs; contributors generate their own and register them.

Graph store. The database that holds the influence graph. Nodes are contributors and artifacts; edges are absorption events with weights, timestamps, channels, and signatures. The graph store must support fast backward traversal (for credit assignment) and efficient appending (for new absorption events).

Attribution engine. The component that computes credit distributions when benefits arrive. Implements spreading activation, Shapley adjustment, and haze decay. Outputs a vector of (contributor_id, amount) pairs for each benefit.

Settlement adapter. The component that connects the attribution engine to payment substrates. Receives credit-distribution vectors and dispatches actual payments via Stripe, Wise, Lightning, etc.

Dispute resolver. The component that handles contested verdicts. Receives dispute filings, routes them to arbitration committees, records rulings, and applies retroactive adjustments.

Audit interface. The component that exposes the algorithm's computation for third-party verification. Anyone can recompute any verdict and verify the result.

Federation gateway. The component that lets this implementation interoperate with other Koin providers. Implements the Federation Protocol Layer (FPL) API. Optional for single-domain pilots; required for cross-domain federation.

The first five components are required for any working Koin implementation. The audit interface is required for federation membership. The federation gateway is required for cross-provider interoperability.

II — Identity (did:koin)

The did:koin DID method is the foundation of Koin's identity layer. Contributors generate their own DIDs locally — no central authority is involved.

Plain English

A DID is like a digital signature paired with a public mailbox address. You create one on your own computer. The private key stays with you; the public part is what others use to verify that you signed something. Your DID can be used across all Koin providers; no single provider owns it. If you switch providers, your DID and your graph history come with you.

Specification

A did:koin DID has the form:

did:koin:<method-specific-id>

Where method-specific-id is a base58-encoded SHA-256 hash of the contributor's public key. The DID resolves to a DID Document containing:

{
  "id": "did:koin:<hash>",
  "publicKey": [...],
  "authentication": [...],
  "service": [
    {
      "id": "<did>#payment-substrates",
      "type": "PaymentSubstrate",
      "serviceEndpoint": {
        "primary": "stripe:cus_...",
        "fallback": ["btc:bc1...", "usdc:0x..."]
      }
    },
    {
      "id": "<did>#preferences",
      "type": "ContributorPreferences",
      "serviceEndpoint": {
        "consentDefault": "explicit",
        "anonymousMode": false,
        "panicContact": "<hashed-email>"
      }
    }
  ]
}

DIDs are signed by their generating key. The DID Document can be updated by signing the update with the same key. Recovery options (in case of key loss) are configurable: time-locked recovery via designated contacts, social recovery, or hardware-key recovery.

Implementation note: do not use vanity DIDs or human-readable identifiers. The cryptographic hash provides identity-independence; human-readable identifiers reintroduce naming hierarchies and create attack surfaces.

III — The graph store

The graph store is the heart of a Koin implementation. It holds every node (contributor or artifact) and every edge (absorption event).

Plain English

Think of a giant scrapbook with photos of every person and every piece of work in your domain. Each photo is connected to other photos by lines that say "this person was influenced by this work" or "this work was based on this earlier work." Each connecting line has a label saying how strong the influence was, when it happened, and what kind of influence it was. The graph store is the structured version of this scrapbook.

Specification

Use a graph database. For pilots up to ~1M nodes: Neo4j or ArangoDB. For larger deployments: distributed graph systems like JanusGraph or TigerGraph. The choice matters less than the schema.

Node schema.

{
  "id": "<did:koin|artifact:hash>",
  "type": "contributor|artifact|institution",
  "metadata": {
    "createdAt": "<ISO-8601>",
    "registeredBy": "<did>",
    "consentScope": "this-pilot|all-pilots|domain-specific"
  },
  "publicKey": "..." // optional, for contributor nodes
}

Edge schema.

{
  "id": "<sha-256-of-edge-content>",
  "source": "<node-id>",
  "sink": "<node-id>",
  "weight": 0.0-1.0,
  "channel": "read|listened|conversed|taught|cited|...",
  "timestamp": "<ISO-8601>",
  "decayRate": 0.0-1.0,  // per year
  "signatures": {
    "source": "<sig>",
    "sink": "<sig>"
  },
  "evidence": {
    "type": "self-reported|inferred|attested",
    "confidence": 0.0-1.0,
    "auditUrl": "..."
  }
}

Edges are append-only. Once written, they are not modified. If new evidence emerges that an edge weight was wrong, a correction edge is appended; the original remains as a record. This is essential for tamper-evidence.

All edges must be cosigned by both source and sink. Edges with missing signatures are rejected by the audit interface and excluded from credit computations.

IV — The attribution engine

When a benefit arrives, the attribution engine computes how much each upstream contributor is owed. This is the computational heart of Koin.

Plain English

A benefit is a moment when value enters the system — someone buys a book, a song is streamed, a paper is cited. The attribution engine asks: who made this benefit possible? It traces backward through the influence graph, finds every contributor whose work led, directly or indirectly, to this benefit, and computes each contributor's share. It runs in milliseconds.

Algorithm

function compute_credits(benefit_node, benefit_amount):
    # Step 1: spreading activation from benefit backward
    credits = spread_credit(
        graph=load_subgraph(benefit_node, max_hops=10),
        sink=benefit_node,
        amount=benefit_amount,
        damping=0.85,
        threshold=0.0001 * benefit_amount,
    )

    # Step 2: Shapley adjustment for non-redundancy
    credits = apply_sampled_shapley(
        credits=credits,
        sample_count=100,
        graph=loaded_subgraph,
    )

    # Step 3: haze decay
    credits = apply_haze_decay(
        credits=credits,
        current_time=now(),
        decay_rates=load_channel_decay_config(),
    )

    # Step 4: reserve commons portion (typically 2%)
    credits, commons_amount = reserve_commons(credits, percentage=0.02)

    # Step 5: dispatch settlement
    return CreditDistribution(
        contributor_amounts=credits,
        commons_amount=commons_amount,
        total=benefit_amount,
        algorithm_version="koin-attr-1.0",
        computed_at=now(),
        verification_hash=compute_hash(credits),
    )

The spreading activation step uses the matrix formulation if the subgraph fits in memory:

credits = (I - d * A.T) ** -1 @ b_vector

Or the iterative variant for larger graphs:

queue = [(benefit_node, benefit_amount)]
while queue:
    node, amount = queue.pop()
    if amount < threshold: continue
    credits[node] += amount
    for edge in incoming_edges(node):
        queue.append((edge.source, amount * damping * edge.weight))

Configurability

The damping factor (default 0.85), the maximum hop count (default 10), the commons percentage (default 2%), and the channel-specific decay rates are all configurable per domain. The configuration is published as part of the audit interface; verifiers can reproduce the algorithm with the published parameters.

Do not tune these parameters in secret. The federation requires that all algorithm parameters be published. A provider that uses private parameters loses federation status.

V — The settlement adapter

Once credits are computed, they need to be paid. The settlement adapter handles the connection between Koin and external payment rails.

Plain English

The attribution engine tells you who should get how much. The settlement adapter actually sends the money. It supports multiple payment substrates (dollars via Stripe, bitcoin via Lightning, etc.) and lets each contributor choose how they want to be paid.

Specification

The settlement adapter implements substrate-specific drivers behind a common interface:

class SettlementDriver:
    def dispatch(self, recipient_did, amount, currency, metadata) -> SettlementReceipt:
        # substrate-specific implementation
        pass

    def check_status(self, receipt_id) -> SettlementStatus:
        pass

    def refund(self, receipt_id, reason) -> SettlementReceipt:
        pass

Implement at minimum:

StripeConnectDriver for USD/EUR/GBP via Stripe Connect. Required for early pilots in regulated economies.

LightningDriver for Bitcoin via the Lightning Network. Required for cross-border micropayments.

USDCDriver for stablecoin payments on Solana, Polygon, or similar low-fee chains. Useful for cross-border without the volatility of Bitcoin.

WiseTransferDriver for traditional cross-border bank transfers. Required for larger payments to contributors in jurisdictions without crypto access.

Optional, for niche pilots:

TimeCreditDriver for time-bank-style settlement (useful in caregiving and educational pilots).

LocalCurrencyDriver for local-exchange-trading-system integration (Brixton Pound, Bristol Pound, etc.).

Settlement is asynchronous. The attribution engine produces a CreditDistribution; the settlement adapter queues each contributor payment for dispatch. Successful dispatches return receipts; failures retry up to N times before being escalated to the dispute resolver.

Batching

Each contributor sets a settlement threshold (default $1.00). Credits below the threshold accumulate; once the threshold is crossed, a batch payment is dispatched. This dramatically reduces transaction fees on rails like Stripe Connect (which charge per transaction) while preserving the perception of continuous payment.

VI — The dispute resolver

Disputes are inevitable. The system must handle them transparently and bindingly.

Plain English

If you think you've been mis-credited — too little, too much, attributed to the wrong work — you file a dispute. A small committee reviews the evidence and makes a binding decision. The decision is public; the algorithm gets better over time as it learns from rulings.

Specification

Disputes are filed via a signed message to the dispute API:

POST /disputes
{
  "disputeId": "<uuid>",
  "filer": "<did>",
  "subject": {
    "type": "credit-distribution|edge-weight|node-attribution",
    "id": "<target-id>"
  },
  "claim": {
    "summary": "...",
    "requestedAdjustment": {...},
    "evidence": [...]
  },
  "signature": "..."
}

The dispute is routed to an arbitration committee of three randomly-selected validators. Validators are contributors with Koin balances above a minimum threshold; selection is weighted by reputation and randomized to prevent capture. Validators have 30 days to render a ruling; rulings are public.

If the ruling adjusts a credit distribution, the system applies retroactive correction:

function apply_retroactive_correction(original_distribution, adjusted_distribution):
    diff = adjusted_distribution - original_distribution
    for contributor_id, amount_change in diff.items():
        if amount_change > 0:
            credit_account(contributor_id, amount_change, reason="dispute-adjustment")
        else:
            debit_account_or_holdback(contributor_id, -amount_change, reason="dispute-adjustment")

Appeals go to a larger federated dispute body (composition determined by federation governance). Federated appeals are rare; the three-person committee handles ~95% of cases.

VII — The audit interface

The audit interface is what makes Koin trustworthy. Anyone, including a third party with no relationship to the implementation, can use it to verify that any verdict was computed correctly.

Plain English

If you don't trust the implementation, you can check its work. The audit interface lets you re-run the algorithm on the same inputs and verify that the outputs match. If they don't, the implementation has a bug or worse — and the federation can hold it accountable.

Specification

The audit interface exposes three endpoints:

GET /audit/algorithm
  Returns the current algorithm version, all parameters, and the open-source code (or a hash of the code with a link to the repository).

GET /audit/verdict/{credit-distribution-id}
  Returns the full computation trace for a given verdict: the subgraph used, the spreading-activation result, the Shapley adjustments, the haze decay, and the final distribution. Anyone can re-run the algorithm with this input and verify the output.

GET /audit/graph-snapshot/{timestamp}
  Returns a Merkle-tree root of the graph state at the given timestamp. Used for tamper-evidence: if the graph state at a past moment differs from this root, the implementation has been tampered with.

Audit endpoints are unauthenticated (rate-limited but otherwise open). Privacy-preserving variants are available for sensitive contributors: the audit can verify the verdict's correctness without revealing the specific contributors involved, using zero-knowledge proofs.

VIII — The federation gateway

The federation gateway lets your implementation interoperate with other Koin providers. Required only for cross-domain federation; optional for single-domain pilots.

Plain English

If your pilot is in education and a contributor's work also got absorbed by people in the music pilot run by another provider, federation lets the two providers share information so the contributor gets credit for both. Federation is what makes Koin a global system rather than a collection of isolated pilots.

Specification

The Federation Protocol Layer (FPL) is an HTTPS API:

GET /fpl/contributors/{did}
  Returns the contributor's portable identity record (public key, payment substrate preferences, consent settings). Used by federated providers to discover that this contributor exists.

GET /fpl/contributions/{did}?domain=<domain>&after=<timestamp>
  Returns a signed list of contributions this provider has recorded for the given contributor in the given domain after the given timestamp. Used by federated providers to incorporate cross-provider contributions into their attribution computations.

POST /fpl/cross-credit
  Submits a cross-provider credit distribution: "we are crediting this contributor for $X, and the trace includes contributions held by your provider." Used to settle cross-provider attribution.

GET /fpl/algorithm-version
  Returns the algorithm version and parameters this provider uses. Used by federated providers to detect incompatibilities.

POST /fpl/dispute-escalation
  Submits a federated dispute (one that cannot be resolved by a single provider's arbitration committee). Routed to the federated dispute body.

Federation membership requires passing periodic compliance audits, publishing algorithm code, and accepting federated rulings. Providers that fail compliance lose federation status; their cross-credits are rejected by other providers.

IX — Deploying a minimum-viable pilot

Here is the minimum sequence for shipping a Koin pilot in a single domain. The total engineering effort is approximately 6-12 weeks for a small team.

Week 1-2: Identity and graph store

Set up the did:koin identity registry. Deploy the graph store with the node and edge schemas above. Build the basic API for registering contributors and recording absorption events. The graph will be empty at this point; you'll populate it in week 3.

Week 3-4: Populate the initial graph

Import the existing relationship data for your domain. For a school district: load the teacher-student-curriculum graph from the student information system. For an open-source project: load the contributor-commit-dependency graph from version control. For a music label: load the artist-collaborator-track graph from existing rights management. Convert each existing relationship into a signed edge with appropriate channel and weight.

Week 5-6: Attribution engine

Implement the attribution algorithm above. Start with the iterative spreading-activation variant (simpler to debug). Add Shapley sampling next. Add haze decay last. Test on synthetic data first; then run dry-passes on the real graph without dispatching settlement.

Week 7-8: Settlement adapter

Integrate with the primary settlement substrate for your domain. For US-based pilots: Stripe Connect. For international: Wise. For crypto-native: Lightning. Test with small amounts to a few contributors before opening to the full pilot population.

Week 9-10: Audit interface and dispute resolver

Implement the audit endpoints. Publish your algorithm code and parameters. Set up the dispute resolver with a small initial committee of trusted validators (you can transition to randomized selection once the validator pool is large enough).

Week 11-12: Soft launch

Open the pilot to a small subset of contributors. Monitor for algorithmic errors, disputes, settlement failures. Iterate. Expand the population gradually as confidence grows.

After week 12, your pilot is operational. Federation can wait until you have a stable single-provider implementation; it should not be the priority for the first pilot.

X — Common pitfalls and how to avoid them

Overfitting the algorithm to your domain. The temptation is to tune the algorithm parameters to produce verdicts that "feel right" for your domain. Resist this. Domain-specific tuning makes your implementation incompatible with federation. The right approach: use the federation defaults, document the cases where the verdicts surprise you, propose parameter changes via the federation governance process.

Treating the graph as authoritative for contestable facts. The influence graph records who-influenced-whom according to recorded evidence. It is not a record of objective truth. Different contributors will remember the same event differently. The system handles this through the dispute resolver, not through algorithmic adjudication. If a contributor disputes an edge, the dispute process resolves it; the algorithm does not make automatic judgments about contested facts.

Underestimating the importance of consent UX. Both sides of every absorption event must consent. The consent process — how it's worded, how it's surfaced, how it's revoked — is the difference between a pilot that grows and a pilot that stalls. Spend disproportionate effort on consent UX. The technical infrastructure is the easy part.

Building the settlement adapter for the wrong substrate. Your contributors will tell you their preferred payment substrate. Listen. A pilot that requires crypto wallets among elderly caregivers will fail. A pilot that requires US bank accounts among international researchers will fail. Match the substrate to the population. Expand later.

Skipping the audit interface. The audit interface seems optional during early development. It is not. Build it before launch. The trust your pilot generates depends on external parties being able to verify your work, and the discipline of building the audit interface forces your implementation to be cleaner than it otherwise would be.

Designing the dispute resolver as an afterthought. The disputes will start in week one of the pilot. Have the dispute resolver ready to handle them on day one. Underestimating dispute volume is one of the most common reasons pilots fail in the early months.

XI — Becoming federation-compatible

If your pilot succeeds and you want to federate, the following steps make your implementation federation-compatible:

Publish your algorithm code and parameters. Open-source the attribution engine. Document every parameter and its rationale. Provide a reproducibility script that takes published verdicts and reproduces them.

Implement the Federation Protocol Layer (FPL) endpoints. The five endpoints described in Section VIII. Test interop with at least one other federated implementation.

Pass a compliance audit. The federation runs periodic compliance audits. These check that your verdict computation matches the published algorithm, that your dispute process is operating, that your graph store is tamper-evident, that you accept federated rulings. Pass it and you are federation-compatible.

Adopt the federation's governance mechanisms. Participate in the algorithm-evolution process. Send representatives to the standards working groups. Implement adopted updates within the required windows.

Federation status is not granted in advance; it is earned by operating compatibly. Once granted, it can be revoked for non-compliance. The federation is a club of well-behaved providers, not a permission system.

XII — Closing remarks for the builder

The most important thing about building Koin is that you are not the first to do it, but you may be the first to do it in your specific domain. The general framework is settled. The specific instantiation in your context — what edges to record, what channels to support, what default decay rates to use, what payment substrate to use, what dispute culture to cultivate — is yours to determine.

Build modestly. Start with the population you know. Solve their problem first. Federate later. The temptation to build the system that handles every case correctly from day one will produce a system that handles no case adequately. The pattern of every successful infrastructure project is the same: solve a specific problem for a specific community, learn from the deployment, generalize incrementally.

The manifesto's ambition is global. The work of getting there is local. Your pilot is local. Build it well. The federation will arrive when enough local pilots are doing well enough that federating them is the obvious next step. That moment is sooner than it looks but later than the manifesto's rhetoric suggests.

Build the smallest thing that works. Ship it. Let the people who use it tell you what to build next. The manifesto is the destination; your pilot is the first step. Take it.
